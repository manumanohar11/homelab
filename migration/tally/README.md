# Tally to ERPNext Migration

This folder is for local Tally export files and generated ERPNext import and cutover files.

The business data folders are gitignored:

```text
migration/tally/raw/
migration/tally/output/
migration/tally/reports/
```

For the current migration there are two separate phases:

1. master import and linking
2. final opening cutover on `1-Apr-2026`

Do not mix the files from those two phases.

Important:

- the current cutover is an opening-balance migration, not a full historical voucher migration
- ERPNext starts from `1-Apr-2026`
- sales bills, purchase bills, receipts, payments, and other voucher documents dated `1-Apr-2025` to `31-Mar-2026` will not appear in ERPNext unless a separate history-import phase is built and run
- Tally remains the source for viewing last year's document-level history unless that separate voucher migration is completed

ERPNext itself can hold multiple fiscal years. For this business, both `2025-2026` and `2026-2027` can exist in the same site. If old invoices are missing, that is a migration-state issue, not an ERPNext fiscal-year limitation.

If you want the previous financial year's documents inside ERPNext too, use the separate full-history track documented in:

1. [`HISTORY_IMPORT_PLAN.md`](HISTORY_IMPORT_PLAN.md)
2. `scripts/tally_build_history_pack.py`
3. `scripts/erpnext_tally_full_history.py`

That history track is reusable for later date ranges too, including the current financial year up to today.

The current history importer stages are:

1. `setup`
2. `opening`
3. `invoices`
4. `settlements`

The `settlements` stage is the step that moves most historical invoices out of
ERPNext `Overdue` status by importing Tally receipts, payments, and journal-like
cash movements after the invoice stage.

## Phase 1: Generate ERPNext Master Import CSVs

From the repo root:

```bash
python3 scripts/tally_to_erpnext.py \
  --company-name "Vara Lakshmi Agencies" \
  --company-abbr "VLA" \
  --warehouse-parent "All Warehouses - VLA"
```

The script writes CSVs under:

```text
migration/tally/output/
```

And review reports under:

```text
migration/tally/reports/
```

Review the generated files before importing anything into ERPNext.

### First Import Order

Use this order in ERPNext Data Import:

1. `uoms.csv`
2. `item_groups.csv`
3. `warehouses.csv`
4. `customers.csv`
5. `suppliers.csv`
6. `items.csv`
7. `addresses.csv` and `contacts.csv` only after deciding how to link them to parties

Do not import these as normal DocTypes:

```text
customer_master_review.csv
supplier_master_review.csv
item_master_review.csv
address_links_staging.csv
contact_links_staging.csv
```

They preserve Tally/source metadata for review and later automation.

## Phase 1A: Link Addresses and Contacts

After customers and suppliers exist in ERPNext, run a dry run:

```bash
python3 scripts/erpnext_import_party_contacts.py
```

If the dry run looks correct, create/update linked Address and Contact records:

```bash
python3 scripts/erpnext_import_party_contacts.py --apply
```

This uses:

```text
addresses.csv
contacts.csv
address_links_staging.csv
contact_links_staging.csv
```

For Warehouse import, `parent_warehouse` must be the existing ERPNext Warehouse document name. In this stack that is currently:

```text
All Warehouses - VLA
```

`All Warehouses` without the company suffix will fail because that exact Warehouse record does not exist.

Important cutover rules:

1. Do not re-import customers, suppliers, items, addresses, or contacts.
2. Do not use the older computed transaction-derived files for the final opening.
3. Do not carry FY `2025-2026` income and expense ledgers into the `1-Apr-2026` opening.
4. Use `Temporary Opening - VLA` as the balancing account during cutover.
5. Use Stock Summary XML totals for opening stock, not the Trial Balance `Opening Stock` row.

## Final Cutover Automation

The repo now includes a host-side ERPNext cutover runner:

```bash
python3 scripts/erpnext_tally_cutover.py
```

What it does:

1. validates the current XML cutover CSV pack
2. checks the live ERPNext site for missing accounts and existing opening documents
3. can create missing accounts
4. can create the opening journal entry
5. can create the opening stock reconciliation
6. can create the temporary-opening clearance journal entry

Safe usage:

```bash
python3 scripts/erpnext_tally_cutover.py
python3 scripts/erpnext_tally_cutover.py --apply
python3 scripts/erpnext_tally_cutover.py --apply --submit
```

- default mode is dry run
- `--apply` creates missing accounts and draft ERPNext documents
- `--apply --submit` creates and submits the selected stage or the whole cutover

You can also run one stage at a time:

```bash
python3 scripts/erpnext_tally_cutover.py --stage accounts
python3 scripts/erpnext_tally_cutover.py --stage journal
python3 scripts/erpnext_tally_cutover.py --stage stock
python3 scripts/erpnext_tally_cutover.py --stage clearance
```

Follow the operator sequence in [`STEP_BY_STEP_OPENING_PROCESS.md`](STEP_BY_STEP_OPENING_PROCESS.md).
